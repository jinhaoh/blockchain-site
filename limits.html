1. 확장성 트릴레마 (Scalability Trilemma)

Vitalik Buterin이 제시한 명제: 탈중앙화 · 보안 · 확장성 중 두 가지만 동시에 달성 가능.

📚 참고: Vitalik: Why sharding is great · Ethereum L2 가이드 · L2BEAT · Celestia: 모듈러

그림. 확장성 트릴레마 — 각 체인의 위치는 어떤 정점을 양보했는지 보여준다

해결 시도

  • Layer 2: Lightning Network (Bitcoin), Optimistic/ZK Rollups (Ethereum)
  • 샤딩(Sharding): 상태와 검증을 분할 처리 (Ethereum Danksharding)
  • 모듈러 블록체인: Celestia 등 데이터 가용성 계층 분리
  • ZK-SNARK / ZK-STARK: 영지식 증명을 통한 압축 검증

2. 기술적 한계

2.1 처리량(TPS) 격차

시스템실측 TPS비고
Bitcoin~710분/블록 ・ 4MB 블록 한계
Ethereum L1~15가스 한도 ・ 12초 슬롯
Solana2,000~5,000이론치 65,000, 노드 요구사항 극히 높음
Ethereum L2 합산10,000+Rollup으로 분산
Visa (참고)24,000 평균 / 65,000 피크중앙화 처리
국내 카드망 (참고)피크 시 수만 TPS중앙화 처리

퍼블릭 블록체인 단독으로는 글로벌 결제망의 처리량을 따라잡기 어렵다. L2 누적·샤딩으로 격차를 좁히는 추세.

2.2 최종성(Finality) 지연

  • Bitcoin: 6 confirm ≈ 약 1시간 (확률적)
  • Ethereum: ~12.8분 (2 epoch, finality gadget)
  • Solana/Cosmos: 1~2초 (결정적)
  • 전통 RTGS (한은금융망 BOK-Wire+): 즉시 결정적 + 법적 효력 명확

2.3 스토리지 증가 (State Bloat)

누적된 상태(account, contract storage)는 모든 풀노드가 보유해야 한다. 이더리움 상태는 2025년 기준 약 1TB+ 규모. 노드 운영자 진입장벽이 높아지며 탈중앙화를 약화시킨다.

  • 해결 시도: Stateless Client, Verkle Tree, Witness 기반 검증

2.4 키 관리 (Key Management)

"Not your keys, not your coins"는 양날의 검이다.
  • 개인키 분실 = 자산 영구 손실 (분실 BTC 추정 300~400만 개)
  • 피싱·악성 컨트랙트 approve로 인한 손실 일상화
  • 오프라인 백업·하드웨어 지갑·MPC·소셜 복구가 표준화 필요

2.5 오라클 문제 (Oracle Problem)

스마트 컨트랙트는 결정론적이라 off-chain 데이터(가격, 날씨, 환율 등)를 직접 가져올 수 없다. 중개자(오라클)가 필요하며, 오라클이 조작되면 컨트랙트 로직이 무력화된다. Chainlink, Pyth, Band Protocol 등이 다중 노드 합의로 완화하지만 근본 해결책은 없다.

2.6 MEV (Maximal Extractable Value)

검증자·채굴자가 트랜잭션 순서를 조작해 이익을 추출하는 현상. 프론트러닝, 샌드위치 공격, 청산 가로채기 등으로 사용자가 손해를 본다. 2021~2024 누적 MEV 추출액 수십억 달러 규모.

  • 완화: Flashbots, MEV-Boost, Encrypted Mempool, CoW Protocol 등

3. 비기술적 한계

3.1 규제 불확실성

  • 증권성 판단(Howey Test 등): SEC vs Ripple 등 다년간 소송
  • 국가 간 규제 격차: 미국 SEC·CFTC 분쟁, EU MiCA 발효, 한국 가상자산이용자보호법(2024.07)
  • AML/CFT: Travel Rule, FATF 권고로 익명성 약화
  • 회계·세무 기준 미성숙 (특히 토큰화 자산)

3.2 에너지 소비 (PoW)

비트코인 연간 전력 소비량은 약 150 TWh (네덜란드 1년 전력 사용량 수준). PoS 전환이 ESG 압력에 대응하는 흐름이지만, 정통성·보안 측면 비판도 존재.

3.3 양자 컴퓨팅 위협

ECDSA/EdDSA 같은 타원곡선 서명은 충분히 강력한 양자컴퓨터가 등장하면 깨질 수 있다(Shor's algorithm). 2030~2040년 사이가 현실적 위협 시점으로 거론되며, NIST 양자내성 알고리즘(ML-KEM, ML-DSA 등) 마이그레이션이 진행 중. 블록체인은 모든 노드가 동시에 업그레이드해야 해서 마이그레이션이 더 어렵다.

3.4 UX 복잡성

  • 지갑 설치·시드구문 보관·니모닉 관리는 일반 사용자에게 부담
  • 가스비 변동, 트랜잭션 실패 시 비용 발생, 주소 입력 오류 등 함정 많음
  • ERC-4337 Account Abstraction이 일부 개선하지만 완성에는 시간 필요

3.5 거버넌스 갈등 (Fork Risk)

중앙 관리자가 없으므로 의사결정 충돌이 체인 분기(fork)로 이어진다. Bitcoin Cash, Ethereum Classic, Solana Saga 등 사례. Code is Law 원칙이 항상 유지되지는 않으며, 사회적 합의가 결정적 변수.

4. 기존 시스템 vs 블록체인 — 분야별 비교

4.1 데이터베이스: PostgreSQL vs 블록체인

특성전통 DB (PostgreSQL 등)퍼블릭 블록체인
처리량수만~수십만 TPSL1: 수~수십 TPS
지연밀리초초~분 단위
저장 비용저렴 (GB당 수 센트)매우 비쌈 (KB당 수 달러)
수정/삭제자유로움 (UPDATE/DELETE)불가능 (append-only)
스키마 변경ALTER TABLE마이그레이션 컨트랙트 필요
신뢰 모델운영 주체 신뢰 필요합의로 변조 방지
가용성운영 주체 책임다수 노드 분산
프라이버시강함 (접근 제어)약함 (퍼블릭은 모두 공개)
감사 가능성로그 의존, 위변조 가능변조 불가능한 영구 기록

4.2 결제 시스템: SWIFT/카드망 vs 블록체인

4.3 자산 등록/등기 시스템: 정부 시스템 vs 블록체인

특성전자등기·정부 시스템블록체인 토큰 (NFT/ERC-3525 등)
법적 효력국가 보증스마트 컨트랙트 ≠ 법적 권리 (현재)
이전 절차등기관 개입, 비용·시간자동·즉시
분할 가능제한적토큰 단위로 자유로움
접근성국가별 시스템 격차인터넷만 있으면 글로벌 접근
강제 회수 (법원)가능ERC-1400처럼 별도 설계 필요

4.4 신원 시스템: 정부 ID vs DID

특성주민등록·정부 ID분산 신원확인(DID)
발급 주체국가다수 발급자 (대학, 회사, 정부 등)
이용자 제어제한적강함 (자기주권신원, SSI)
선택적 공개전부 또는 전무나이 ≥ 19세만 증명 등 가능 (ZK)

호환성국가별 단절W3C VC/DID 표준으로 글로벌 호환

5. 블록체인이 적합한 경우 vs 부적합한 경우

✅ 블록체인이 정말 필요한 경우

  • 다수 당사자가 공동 신뢰 기관 없이 협력: 컨소시엄, 국경간 결제
  • 변조 저항성이 결정적: 공증, 학위 검증, 공급망 이력
  • 토큰화·프로그래머블 가치 이전: CBDC, 증권 토큰, DeFi
  • 최종 사용자가 자산을 자기 보관하는 것이 본질: 자기주권신원, 검열저항
  • 스마트 컨트랙트로 신뢰 비용 절감: 에스크로, 자동 배당

❌ 블록체인이 부적합한 경우

  • 단일 조직이 데이터를 통제: 그냥 DB가 낫다
  • 고빈도·저지연이 필수: 거래소 매칭엔진 같은 ms 단위
  • 개인정보 등 비공개 데이터 저장: GDPR 위배 가능, 오프체인 + 해시 앵커링 권장
  • 잦은 수정·삭제 필요: 블록체인은 append-only
  • 참여자가 신뢰 가능한 소수: 분산 합의 비용이 무의미
  • 저장량이 매우 큰 데이터: 영상·이미지는 IPFS·S3 + 해시만 온체인

의사결정 플로우

Q1. 여러 당사자가 공동 신뢰 기관 없이 데이터를 공유해야 하는가?
    └─ NO  → 전통 DB 사용 (블록체인 불필요)
    └─ YES → Q2로

Q2. 모든 참여자가 같은 데이터를 봐야 하는가?
    └─ NO  → 메시징 시스템 + 개별 DB 충분
    └─ YES → Q3로

Q3. 데이터의 무결성·이력이 법적·규제적으로 중요한가?
    └─ NO  → 공유 DB로 충분 (블록체인 과잉)
    └─ YES → Q4로

Q4. 참여자 신원을 통제할 수 있는가?
    └─ YES → 퍼미션드 DLT (Fabric, Corda, Quorum)
    └─ NO  → 퍼블릭 블록체인 (Ethereum, L2)

Q5. 자산이나 가치를 토큰으로 표현해야 하는가?
    └─ YES → 스마트 컨트랙트 + 토큰 표준 (ERC-20/721/3525/1400 등)
    └─ NO  → 단순 기록용이면 머클 앵커링도 고려

6. 실패·왜곡 사례에서 배우기

사례문제의 본질교훈
Walmart Hyperledger Pilot 실효성 입증 못 함, 운영 비용 증가 중앙 관리자가 있는 환경에 블록체인 강제 시 오버엔지니어링
Australia ASX CHESS 교체 실패 (Digital Asset 활용) 2022년 약 2.5억 호주달러 손실, 프로젝트 중단 레거시 인프라 교체는 기술 외 거버넌스·테스트가 더 중요
Terra/LUNA 붕괴 (2022.05) 알고리즘 스테이블코인의 죽음의 나선 경제 모델·게임이론 결함 → 기술이 견고해도 시스템 붕괴
FTX 파산 (2022.11) 중앙화 거래소의 고객 자산 횡령 블록체인이 아무리 투명해도 CEX는 별개 — "Not your keys"
NFT 가격 폭락 (2022~2023) 유틸리티 없는 투기 → 90%+ 손실 토큰 기술 ≠ 비즈니스 모델

7. 한국은행·금융권 관점의 종합 평가

중앙은행과 금융권의 의사결정 관점에서 본 블록체인의 위치:

  • 퍼블릭 블록체인은 결제 인프라로 부적합: TPS, 최종성, 규제, 법적 효력에서 한계
  • 퍼미션드 DLT는 효과적: 컨소시엄·국제송금·도매 결제에 적합 (Fabric, Corda, mBridge)
  • 예금토큰 모델은 유망: 한강 프로젝트가 보여주듯 기존 통화시스템과 정합성 유지
  • 토큰화 자산(STO)의 잠재력: ERC-3525·ERC-1400은 채권·증권 토큰화에 적합
  • 스테이블코인의 양면성: 효율 ↑, 통화주권·금융안정성 위협 가능 → CBDC 필요성 증대
  • x402 같은 에이전트 결제 표준: M2M 경제의 새 결제 인프라 가능성, 통화·결제 정책에 시사점

요약: 블록체인을 "범용 솔루션"으로 보기보다, 특정 마찰 비용을 줄이는 도구상자의 한 칸으로 인식하고 유스케이스별 적합성을 판단하는 것이 실무적으로 효과적이다.
