x402 소개

x402는 1997년 HTTP/1.1 명세에서 예약만 되어 있던 HTTP 402 Payment Required 상태 코드를 실제로 작동하게 만든 오픈 결제 프로토콜이다. 2025년 Coinbase가 공개했으며, AI 에이전트와 머신 간 결제(machine-to-machine), API 종량 과금을 1차 타깃으로 한다.

📚 참고: x402.org 공식 · coinbase/x402 GitHub · EIP-3009 · EIP-712 · RFC 7231 §6.5.2 · Circle USDC

1. 마이크로페이먼트와 블록체인

x402의 등장 배경을 이해하려면 먼저 마이크로페이먼트(Micropayment)가 무엇이고 왜 전통 결제 시스템에서는 불가능했는지를 살펴볼 필요가 있다.

1.1 마이크로페이먼트란?

일반적으로 $0.01 ~ $1 (수십원 ~ 천원) 수준의 소액 결제를 의미한다. 더 작은 단위($0.001 미만)는 나노페이먼트(Nanopayment)로 부르기도 한다.

1.2 왜 전통 결제망에서는 안 되는가?

구조적으로 불가능
전통 결제망은 "큰 결제"에 최적화되어 마이크로페이먼트 시장을 외면해왔다

1.6 마이크로페이먼트의 한계

  • L1 가스비: Ethereum L1에서는 마이크로페이먼트도 수 달러 가스비 — L2 이주 필수
  • UX 복잡성: 지갑 설정·잔액 충전·서명 확인이 일반 사용자에게 부담 — AA(Account Abstraction)·세션키로 개선 진행
  • 회계·세무: 1만 번의 $0.01 결제를 어떻게 회계 처리할 것인가
  • 규제: 가상자산 결제의 법적 지위, 자금세탁방지(AML), 한국의 가상자산이용자보호법 적용 범위
  • 거래소 환금성: 마이크로페이먼트로 받은 토큰을 일상 화폐로 바꾸는 비용·시간
  • 가격 변동성: 스테이블코인을 쓰지 않으면 결제 후 가치 변동 위험
요점: 마이크로페이먼트는 새로운 비즈니스 모델(에이전트 경제·콘텐츠 직접 보상)을 가능하게 하는 핵심 기반이지만, 블록체인만으로는 부족하다. L2 확장성, 스테이블코인의 가치 안정, AA의 UX 개선, 규제 정합성이 모두 갖춰져야 실용 단계에 진입한다. x402는 이 모든 조건을 종합한 표준화 시도의 하나로, 아래에서 자세히 다룬다.

2. 왜 지금인가?

  • AI 에이전트 경제: 자율 에이전트가 API·콘텐츠·연산 자원을 마이크로결제로 소비
  • 스테이블코인의 정착: USDC 등 결제 적합성·결제 최종성·낮은 수수료를 갖춘 자산 등장
  • 전통 결제망의 한계: 신용카드 최소 결제 단위($0.30+α), KYC, 30~120일 지연 정산
  • 표준의 부재: API 키 + 청구서 모델이 에이전트 워크플로우에 부적합

4. EIP-3009: 가스리스 결제의 핵심

x402가 사용자 친화적인 이유는 EIP-3009 transferWithAuthorization에 있다. 사용자(에이전트)는 USDC 자체만 보유하면 되고, ETH 가스비는 facilitator가 대납한다. 서명은 EIP-712 typed data로 만들어진다. 

// USDC 컨트랙트의 EIP-3009 인터페이스
function transferWithAuthorization(
    address from,
    address to,
    uint256 value,
    uint256 validAfter,
    uint256 validBefore,
    bytes32 nonce,
    uint8 v, bytes32 r, bytes32 s   // EIP-712 서명
) external;

// 클라이언트(에이전트) 측 의사 코드
const auth = {
  from: agentWallet.address,
  to: serverPaymentAddress,
  value: parseUnits("0.001", 6),     // USDC 6 decimals
  validAfter: 0,
  validBefore: now() + 600,          // 10분 유효
  nonce: randomBytes32(),
};
const signature = await agentWallet.signTypedData(
  USDC_DOMAIN, EIP3009_TYPES, auth
);
// → X-PAYMENT 헤더로 전송

5. facilitator 패턴

서버가 매 요청마다 직접 온체인 검증·정산을 하면 RPC 비용·지연이 발생하므로, facilitator라는 중개자가 권장된다.

구성요소역할
Client / Agent지불 의사 결정 + EIP-3009 서명
Resource Server402 응답으로 가격 통지 + 결제 후 자원 반환
Facilitator서명 검증, 온체인 정산 대납, txHash 반환
Settlement NetworkBase, Ethereum, Solana 등 결제 자산이 발행된 체인

6. 시나리오: 에이전트가 자율적으로 API 결제

// LLM 에이전트가 유료 검색 API를 호출하는 의사 코드
async function callPaidAPI(url, agentWallet) {
  let res = await fetch(url);

  if (res.status === 402) {
    const requirement = JSON.parse(res.headers.get('X-PAYMENT-REQUIRED'));

    // 에이전트의 정책 (예산, 자산, 화이트리스트) 검사
    if (!agentPolicy.canPay(requirement)) {
      throw new Error("Payment policy rejected");
    }

    const payment = await signX402Payment(requirement, agentWallet);
    res = await fetch(url, {
      headers: { 'X-PAYMENT': payment }
    });
  }
  return res.json();
}

7. x402의 한계

  • 규제·세무 모호성: 자율 에이전트의 결제 주체는 누구인가? 한국은 가상자산이용자보호법 적용 검토 필요
  • 가격 발견 메커니즘 부재: 정적 가격 책정 → 동적 경매·번들링은 후속 표준 영역
  • 리플레이·재정산 공격: nonce + validBefore에 의존 → 클라이언트 구현 결함 시 이중지불 위험
  • facilitator 신뢰: 단일 facilitator는 검열 위험, 다중화 시 비용 증가
  • 비-EVM 체인 확장: 현재 USDC on Base가 1차 타깃, Solana/L2 확장은 진행 중
  • 금융권 시각: CBDC·예금토큰을 결제 자산으로 사용할 수 있는가? 한강 모델과 결합 시 새로운 통화정책 함의 발생
한국은행 관점의 함의: x402가 가정하는 결제 자산은 현재 스테이블코인이지만, 프로토콜 자체는 자산 중립적이다. 한강 프로젝트의 예금토큰을 x402 결제 자산으로 사용하는 시나리오는, M2M·에이전트 경제에서 은행화폐(commercial bank money)의 단위성 유지를 실험할 수 있는 흥미로운 정책 실험이 될 수 있다.

8. AP2 (Agent Payments Protocol) — 에이전트 commerce의 상위 표준

x402가 "결제 실행 계층"이라면, AP2는 그 위의 "권한·위임·라우팅 계층"이다. Google이 2025년 9월 공개한 오픈 프로토콜로, AI 에이전트가 사용자를 대신해 안전하게 상거래를 수행할 수 있도록 표준화된 Mandate(위임) 체계를 정의한다.

📚 참고: google-agentic-commerce/AP2 GitHub · Google 발표 · AP2 v0.1 명세

8.1 왜 AP2인가? — x402만으로는 부족한 부분

x402는 강력한 마이크로결제 인프라이지만, 다음과 같은 질문에 답하지 않는다:

  • 사용자가 에이전트에게 어디까지 권한을 위임했는가?
  • 에이전트가 구매하려는 상품이 사용자 의도와 일치하는가?
  • 여러 결제수단(카드·은행·크립토) 중 어떤 것을 쓸지 누가 결정하는가?
  • 분쟁 발생 시 책임 추적이 가능한가?

AP2는 이 질문들에 답하는 "에이전트 commerce 거버넌스 표준"이다.

8.2 핵심 개념 — 3종 Mandate

AP2의 핵심은 암호학적으로 서명된 위임장(Mandate) 체계다. 사용자 → 에이전트 → 판매자 → 결제 인프라로 전달되며, 각 단계의 권한과 책임을 명확히 분리한다.

8.3 AP2 + x402 + CBDC 스택 구조

AP2 = 상위 권한·라우팅 계층, x402·카드·은행·CBDC = 하위 결제 실행 계층

두 프로토콜은 경쟁이 아니라 보완 관계다. 계층이 다르다.

     },
     "expires_at": "2026-07-30T00:00:00Z",
     "signature": "0xabc...(user signature)"
   }

② Agent가 항공권 검색·선택 후 → Cart Mandate (에이전트 ↔ 판매자)
   {
     "merchant": "did:web:korean-air.example",
     "items": [{
       "flight": "KE1234 ICN→PUS 2026-07-05",
       "price": "85000 KRW"
     }],
     "agent_signature": "0xdef...",
     "merchant_signature": "0x123..."
   }

③ Payment Mandate (에이전트 → 결제 인프라)
   {
     "payment_method": "x402",        // 또는 "card", "bank", "cbdc"
     "asset": "USDC",
     "amount": "65 USDC",             // 85,000 KRW 환산
     "recipient": "0xairline...",
     "user_authorization": "ref to Intent Mandate",
     "signature": "0x789..."
   }

④ x402가 실제 결제 실행 → 항공권 발행 → 사용자에게 알림

8.6 AP2의 핵심 기술 요소

  • W3C Verifiable Credentials: 디지털 신원·자격증 검증 표준 활용
  • DIDs (Decentralized Identifiers): 사용자·에이전트·판매자 식별
  • JSON-LD: Mandate 데이터 표현 형식
  • Cryptographic Signatures: ECDSA·EdDSA 등으로 각 Mandate 서명
  • Audit Trail: Mandate 체인이 분쟁 시 책임 추적 증거

8.7 한국은행·금융권 시사점

AP2 + x402 + 한강 예금토큰의 결합 시나리오:

  1. AI 에이전트의 자율 결제가 표준화된 권한 위임(AP2)으로 안전하게 가능해지면, 한국 일반 사용자가 한강 예금토큰을 결제 수단으로 채택하는 자연스러운 경로 형성
  2. 중앙은행 화폐(예금토큰) → AP2 결제 수단으로 등록 → AI 에이전트 경제에서 "은행화폐 단위성" 유지 가능
  3. 규제 친화적: AP2의 Mandate 체계는 KYC·AML·소비자 보호 요건과 자연스럽게 결합
  4. BIS Agora와의 연계: AP2를 통한 국경간 에이전트 결제에서 Agora 통합원장이 정산 백엔드로 활용 가능

정책 함의: 한국은행이 한강 예금토큰을 AP2 결제 수단으로 등록·표준화하는 작업은, 향후 AI 에이전트 commerce 시대에서 스테이블코인이 결제 인프라를 독점하는 것을 방지하는 예방적 정책 옵션이 될 수 있다.

8.8 AP2의 한계

  • 표준화 초기: 2025.09 공개 후 진화 중, 사양 변경 가능성
  • Google 주도 거버넌스: 진정한 다자 표준이 될지 시간 필요
  • 경쟁 표준 가능성: Apple·Amazon·Microsoft 등의 별도 표준 등장 가능
  • 법적 책임 미정비: 에이전트의 잘못된 거래에 대한 책임 분담 (사용자·에이전트·플랫폼) 법적 정리 필요
  • 프라이버시: Mandate 체인이 거래 이력을 노출 → 선택적 공개 메커니즘 필요
  • 실제 채택률: 명세 발표와 실제 보급 사이의 격차 (어떤 기업이 먼저 채택할 것인가)
요약: AP2는 "에이전트 commerce의 거버넌스 표준", x402는 "블록체인 결제 실행 인프라"로 역할 분리. 두 표준이 결합되면 AI 에이전트가 안전하고 규제 친화적인 자율 commerce를 수행할 수 있게 된다. 한국은행은 한강 예금토큰을 이 스택의 결제 수단으로 자리매김하는 옵션을 검토할 만한 시기.
