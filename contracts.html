1. 트랜잭션 생명주기

  1. 생성: 클라이언트가 트랜잭션 구성 → privKey로 ECDSA/EdDSA 서명
  2. 전파: P2P gossip 프로토콜로 메모리풀(Mempool)에 진입
  3. 검증: 노드들이 서명·잔액·논스 검증
  4. 블록 포함: 채굴자/검증자가 수수료 기준 우선순위로 선택
  5. 합의: 합의 알고리즘에 따라 블록 확정
  6. 최종성 도달: 비트코인 6 confirm (≈ 1시간), 이더리움 finality gadget

2. 스마트 컨트랙트 개요

블록체인 위에서 실행되는 결정론적 프로그램. 이더리움의 EVM(Ethereum Virtual Machine)이 대표적이며, Solidity로 작성된 코드는 바이트코드로 컴파일되어 모든 노드에서 동일하게 실행된다.

📚 참고: Solidity · Ethereum: EVM · OpenZeppelin · SWC Registry · Foundry Book

EVM의 핵심 개념

  • Stack-based VM: 1024 깊이 스택, 256-bit 워드 (Big Number 친화)
  • Storage: 영구 저장, 슬롯(256-bit) 단위, 가장 비싼 자원
  • Memory: 트랜잭션 동안만 유지, 선형 메모리
  • Gas: 모든 연산에 비용 부과 → DoS 방지, 우선순위 시장
  • Determinism: 동일 입력 → 동일 출력 (블록 hash, gaslimit 등 환경 변수 제외)

기본 Solidity 컨트랙트 예시

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Counter {
    uint256 public count;
    address public immutable owner;

    event Incremented(address indexed by, uint256 newCount);

    error NotOwner();

    constructor() {
        owner = msg.sender;
    }

    modifier onlyOwner() {
        if (msg.sender != owner) revert NotOwner();
        _;
    }

    function increment() external onlyOwner {
        unchecked { count++; }   // 0.8.x: 명시적 unchecked로 가스 절약
        emit Incremented(msg.sender, count);
    }
}

3. 주요 토큰 표준 (ERC Standards)

ERC(Ethereum Request for Comments)는 이더리움 생태계의 합의된 인터페이스 표준이다. 토큰의 동작 방식을 표준화하여 지갑·DEX·NFT 마켓플레이스 등이 호환되게 한다. 아래는 가장 중요한 9개 표준을 심층 분석한다.

📚 참고: EIPs Index (ERC 전체 목록) · OpenZeppelin 구현

ERC-20 Fungible Token Standard Final (2015.11)

대체 가능 토큰의 사실상 표준. USDC, USDT, DAI, UNI 등 거의 모든 메인 토큰이 ERC-20. 각 토큰 1개가 다른 1개와 완벽히 동일한 가치를 가진다는 가정 위에 동작한다.

핵심 인터페이스

interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}

실제 사용 사례

  • 스테이블코인: USDC, USDT, DAI
  • 거버넌스 토큰: UNI, COMP, AAVE
  • 유틸리티 토큰: LINK, GRT
  • WETH: ETH를 ERC-20으로 래핑 (DeFi 호환용)
유명한 함정 — Approve Race Condition: approve(spender, X)를 approve(spender, Y)로 변경할 때, 공격자가 그 사이에 transferFrom으로 X를 빼간 뒤 Y도 빼갈 수 있다. 해결: increaseAllowance/decreaseAllowance 사용 또는 ERC-2612 Permit으로 대체.

📚 EIP-20 원문 · OpenZeppelin ERC20

ERC-721 Non-Fungible Token (NFT) Final (2018.01)

대체 불가능 토큰 표준. 각 토큰은 고유한 tokenId를 가지며, 1:1로만 소유된다. 디지털 아트(CryptoPunks, BAYC), 도메인(ENS), 게임 아이템, 부동산 등 유일성이 중요한 자산에 적합.

핵심 인터페이스

interface IERC721 {
    function balanceOf(address owner) external view returns (uint256);
    function ownerOf(uint256 tokenId) external view returns (address);
    function safeTransferFrom(address from, address to, uint256 tokenId) external;
    function transferFrom(address from, address to, uint256 tokenId) external;
    function approve(address to, uint256 tokenId) external;
    function setApprovalForAll(address operator, bool approved) external;
    function getApproved(uint256 tokenId) external view returns (address);
    function isApprovedForAll(address owner, address operator) external view returns (bool);
}

interface IERC721Metadata {
    function name() external view returns (string memory);
    function symbol() external view returns (string memory);
    function tokenURI(uint256 tokenId) external view returns (string memory);  // 메타데이터 JSON URL
}

ERC-1155 Multi-Token Standard Final (2019.06)

Enjin이 게임 아이템을 위해 제안한 다중 토큰 표준. 단일 컨트랙트로 대체 가능 토큰과 NFT를 동시에 관리할 수 있어, 게임처럼 수천 종의 아이템을 다루는 데 최적. 배치(batch) 전송으로 가스를 크게 절약한다.

핵심 인터페이스

interface IERC1155 {
    function balanceOf(address account, uint256 id) external view returns (uint256);

    function balanceOfBatch(
        address[] calldata accounts,
        uint256[] calldata ids
    ) external view returns (uint256[] memory);

    function setApprovalForAll(address operator, bool approved) external;
    function isApprovedForAll(address account, address operator) external view returns (bool);

    function safeTransferFrom(
        address from, address to,
        uint256 id, uint256 amount,
        bytes calldata data
    ) external;

    function safeBatchTransferFrom(    // ← 한 트랜잭션으로 여러 토큰 전송
        address from, address to,
        uint256[] calldata ids,
        uint256[] calldata amounts,
        bytes calldata data
    ) external;
}

ERC-3525 Semi-Fungible Token (SFT) Final (2022.09)

금융 자산 토큰화를 위해 설계된 차세대 표준. ERC-721처럼 고유 tokenId를 가지지만, ERC-20처럼 quantitative value를 함께 갖는다. 여기에 slot이라는 그룹 개념을 추가해, 같은 slot의 토큰끼리는 가치 단위가 호환된다.

3차원 데이터 구조

┌─────────────────────────────────────────────┐
│  ERC-721:    [ tokenId ]                    │  유일성만
│  ERC-20:     [ value ]                      │  수량만
│  ERC-3525:   [ tokenId, slot, value ]       │  ← 셋 다
└─────────────────────────────────────────────┘

예시: 채권 토큰화
  tokenId = 1   slot = "2030년 만기 회사채 A"   value = 1,000,000원
  tokenId = 2   slot = "2030년 만기 회사채 A"   value = 500,000원
  tokenId = 3   slot = "2026년 만기 국채 B"     value = 3,000,000원

  → tokenId 1과 2는 같은 slot이므로 가치 합산·분할·전송 가능
  → tokenId 3은 다른 slot이므로 1·2와 직접 호환 안 됨

핵심 인터페이스

interface IERC3525 {
    // slot 조회
    function slotOf(uint256 tokenId) external view returns (uint256);

    // 토큰의 잔액(가치)
    function balanceOf(uint256 tokenId) external view returns (uint256);

    // 같은 slot 토큰 간 가치 일부 전송 (분할 가능)
    function transferFrom(
        uint256 fromTokenId,
        uint256 toTokenId,
        uint256 value
    ) external;

    // 새 tokenId로 가치 분리 (NFT를 쪼개기)
    function transferFrom(
        uint256 fromTokenId,
        address to,
        uint256 value
    ) external returns (uint256 newTokenId);
}

적용 사례 — 금융권 관점에서 중요

  • 채권 토큰화: 만기·이자율 같은 slot, 액면금액은 value로 → 부분 양도 가능
  • 옵션·선물: 만기·행사가 동일 slot, 계약 수량 value
  • 베스팅 스케줄: 미수령분의 분할·이전
  • 탄소배출권: 발행연도·기준 slot, 톤 수 value
  • STO(Security Token Offering): 일물일가 + 분할 거래
  • CBDC 예금토큰 응용 가능성: 발행기관·만기를 slot으로, 액면을 value로 — 차등 통화 정책 도구 실험
왜 중요한가: 기존 표준은 금융상품의 "같은 종류이지만 액면금액이 다른" 자산을 표현하지 못했다. ERC-3525는 1개 컨트랙트로 동일 상품군의 다양한 액면 단위를 자연스럽게 다룬다. 한국의 토큰증권(STO) 가이드라인과도 친화적이다.

📚 EIP-3525 원문 · Solv Protocol (3525 활용 예시)

ERC-2612 Permit Extension for ERC-20 Final (2020.04)

ERC-20의 approve() 트랜잭션을 오프체인 서명으로 대체. 사용자가 가스를 내지 않고 승인 의사를 표현할 수 있어 DeFi UX 혁신의 핵심. DAI가 처음 도입했고 현재 USDC도 지원. 

function permit(
    address owner,
    address spender,
    uint256 value,
    uint256 deadline,
    uint8 v, bytes32 r, bytes32 s
) external;

// 사용자는 EIP-712 서명만 만들고, 누구든 그 서명을 들고 permit 호출 가능
// → 사용자가 ETH(가스) 없이도 approve + transferFrom 가능

📚 EIP-2612

ERC-4337 Account Abstraction Final (2023.03)

지갑 자체를 스마트 컨트랙트로 만든다. 기존 EOA(외부 소유 계정)와 달리, 서명 방식·복구·결제 자산을 자유롭게 정의 가능. 프로토콜 변경 없이 별도 mempool로 구현.

주요 구성 요소

컴포넌트역할
UserOperation일반 트랜잭션과 분리된, AA 전용 트랜잭션 객체
EntryPoint온체인 검증·실행 엔진 (싱글톤 컨트랙트)
BundlerUserOp를 모아 일반 트랜잭션으로 묶어 EntryPoint에 제출
Paymaster가스비 대납 (스폰서십, USDC로 가스 결제 등)
Smart Account사용자 지갑 자체가 컨트랙트

가능해지는 것들

  • 소셜 로그인: 이메일·Google OAuth로 지갑 생성
  • 키 분실 복구: 사회적 복구, 멀티시그
  • 일괄 트랜잭션: approve + swap을 한 번에
  • 대납 결제: 사용자가 USDC만 갖고도 가스 없이 거래
  • 세션 키: 게임에서 매번 서명 없이 일정 시간 자동 거래

📚 EIP-4337 · Alchemy AA 가이드 · Stackup (Bundler)

ERC-4626 Tokenized Vault Standard Final (2022.03)

DeFi의 수익 발생 볼트(yield vault)를 표준화. Yearn, Aave 등에서 사용하는 "예금하면 다른 토큰을 받고, 수익률에 따라 환산 비율이 변하는" 모델을 단일 인터페이스로 정의. 

interface IERC4626 is IERC20 {
    function asset() external view returns (address);            // 예금 자산
    function totalAssets() external view returns (uint256);

    function convertToShares(uint256 assets) external view returns (uint256);
    function convertToAssets(uint256 shares) external view returns (uint256);

    function deposit(uint256 assets, address receiver) external returns (uint256 shares);
    function mint(uint256 shares, address receiver) external returns (uint256 assets);
    function withdraw(uint256 assets, address receiver, address owner) external returns (uint256);
    function redeem(uint256 shares, address receiver, address owner) external returns (uint256);
}

활용: 유동성 풀, lending 프로토콜, MEV 보호 볼트, 인덱스 펀드 등.

📚 EIP-4626

ERC-6551 Token Bound Accounts (TBA) Final (2023.05)

NFT 자체가 지갑 주소를 갖는다. 모든 ERC-721 NFT에 결정론적으로 매핑되는 스마트 컨트랙트 계정을 부여해, NFT가 다른 토큰·NFT를 보유하거나 트랜잭션을 발신할 수 있게 한다.

활용 시나리오

  • 게임 캐릭터 NFT가 무기·아이템(다른 NFT)을 인벤토리에 보유
  • 아바타 NFT가 자체 잔액·이력을 누적 (소셜 그래프)
  • NFT를 양도하면 그 안의 자산까지 통째로 이전
  • 부동산 NFT가 임대료 USDC를 자동 수령
// 결정론적 주소 생성
function account(
    address implementation,
    bytes32 salt,
    uint256 chainId,
    address tokenContract,
    uint256 tokenId
) external view returns (address);
// → 같은 NFT는 항상 같은 주소를 가짐

ERC-1400 Security Token Standard (비공식) 정식 EIP 미채택

규제 친화적 증권형 토큰 표준 제안. 발행자가 KYC·AML 요건에 따라 이전을 제어할 수 있고, 문서 첨부, 강제 이전(법원 명령 등), 부분(partition) 단위 발행 등을 지원. ※ 다만 ERC-1400은 ethereum/EIPs에 정식 채택되지 않은 비공식 표준 묶음으로, 현장에서는 각 발행자가 일부 기능만 채택해 구현하는 경우가 많다.

  • canTransfer(): 이전 사전 검증 (KYC 통과 여부, 락업 등)
  • operatorRedeemByPartition(): 규제 당국의 강제 회수
  • setDocument(): 발행 약관·정관 첨부

4. 자산 토큰화 vs 기존 디지털화 — 무엇이 다른가?

종이 증서나 실물 자산을 컴퓨터로 옮긴 "디지털화"와, 블록체인 위의 토큰으로 표현하는 "토큰화"는 겉으로는 비슷해 보이지만 본질이 다르다. 이 차이를 이해하는 것이 STO·CBDC·RWA(실물자산) 시장을 이해하는 핵심이다.

4.1 정의 비교

디지털화는 "기관이 보관하는 전자 기록", 토큰화는 "본인이 직접 보유하는 암호학적 자산"

4.2 핵심 차이점 — 9가지 측면

측면디지털화 (전자 기록)토큰화 (블록체인 토큰)
보관 주체 중앙기관 (예탁결제원·은행·증권사) 분산 원장 (사용자 또는 위탁 지갑)
소유 증명 기관의 장부 기록 = "내 잔액" 개인키 보유 = "내 자산"
이전 메커니즘 기관 시스템 경유 (DvP·중개) P2P 즉시 (스마트 컨트랙트)
분할 가능성 액면가·최소 단위 제약 자유로움 (특히 ERC-3525)
운영 시간 영업시간·결제일 제약 24/7/365
프로그래머블 제한적 (배치 처리) 완전 — 자동 배당·조건부 이전·DvP
글로벌 접근성 국가·기관별 단절 인터넷만 있으면 무제한
법적 효력 명확 (법령으로 보장) 진화 중 (국가별 상이)
검열·강제 회수 기관·법원이 가능 설계에 따라 다름 (ERC-1400은 가능)

4.4 토큰화가 제공하는 새로운 가치 (잠재적)

아래는 토큰화의 잠재적 가치이다. 실제 실현 여부는 규제·시장 수요·인프라 성숙도에 따라 달라진다. 예를 들어 "유동성 증대"는 토큰화한다고 자동으로 생기지 않고 충분한 수요가 필요하며, "글로벌 접근"은 KYC·자본통제 등으로 제약된다.

  • 분할(Fractionalization): 고가 자산을 작은 단위로 → 빌딩 1채 → 1만 토큰으로 1만 명이 공동 투자
  • 유동성 증대: 비유동 자산(부동산·미술품·비상장주)을 거래 가능한 토큰으로
  • 24/7 거래: 시간대·휴일 제약 해소
  • 자동화 (Composability): 배당·이자·환매 등이 코드로 자동 실행
  • DvP/PvP 동시이행: 결제와 자산이전이 atomic하게 결합 → 결제실패 위험 0
  • 글로벌 접근: 국경·기관 제약 없는 투자자 풀
  • 투명한 이력: 모든 거래·소유권 변동이 영구 기록
  • 프로그래머블 권리: 일정 조건 충족 시 자동 권리 행사 (예: 베스팅 종료 시 자동 이전)

4.5 토큰화의 한계

  • 법적 정합성: "토큰을 소유하면 법적으로 자산을 소유한다"는 명확한 법령 정비 진행 중
  • 오라클·신뢰 문제: 부동산 토큰은 실제 등기와 어떻게 연결되는가? (Off-chain ↔ On-chain 연결)
  • 강제 회수의 어려움: 법원 명령으로 자산 동결·환수가 디지털화에서는 쉽지만 토큰은 별도 설계 필요
  • UX·키 관리: 일반 사용자에게 개인키 관리는 여전히 큰 부담
  • 회계·세무 처리: 기존 시스템과의 정합성, 평가·과세 기준 미성숙
  • 운영 비용: 가스비, 노드 운영, 보안 감사 등이 추가 비용
  • 규제 리스크: 증권성 판단, AML/CFT, 국경간 규제 갈등

4.6 기존 시스템 대비 진짜 가치 — 객관적 재평가

앞서 나열한 8가지 토큰화의 가치는 매력적으로 들리지만, 대부분은 기존 시스템에서도 어느 정도 제공 가능하다. 토큰화의 독점적 가치는 생각보다 좁다. 정직하게 항목별로 재평가한다.

토큰화 주장기존에서 가능?객관적 평가
분할 (Fractionalization) ✅ 가능 REITs, ETF, 뮤추얼펀드, K-OTC, 토스 1주 미만 투자 — 이미 분할투자 보편화. 토큰화는 분할 단위를 더 작게 할 수 있을 뿐
유동성 증대 ⚠️ 부분적 상장 REIT는 매우 유동적. 유동성은 토큰화로 생기지 않고 수요가 만든다. 실제 토큰화 부동산 대부분 거래량 미미
24/7 거래 ✅ 국내 가능 한국 카카오뱅크/토스는 사실상 24/7. BOK-Wire+ 실시간. 국제 거래는 토큰화 우위, 국내는 차이 작음
자동화 (Composability) ✅ 가능 자동이체, ACH, 정기지급, 자동배당 모두 이미 존재. 토큰화는 더 유연하지만 "새로 가능한" 것은 아님
DvP/PvP 동시이행 ✅ 가능 CLS(FX), 예탁결제원 DvP, T+2 결제 작동 중. 토큰화는 결제일을 T+0으로 단축하는 정도
글로벌 접근 ⚠️ 제약 있음 SWIFT가 글로벌. 외환관리법·자본통제로 토큰화도 결국 KYC 필요. 법적으로 완전 자유롭지 않음
투명한 이력 ✅ 가능 금감원 감사, 거래소 공시, 회계 감사 모두 이력 추적. 퍼블릭 체인 외에는 차이 작음
프로그래머블 권리 ⚠️ 제한적 에스크로, 트러스트, 법적 계약으로 가능. 토큰화는 유연성·속도·자동화 수준에서 우위
자주 과장되는 주장들:
  • "중개자 없이 결제" → 사실은 거래소·지갑·오라클·브릿지 등 새 중개자가 생김
  • "수수료 0" → L1 가스비, L2 sequencer, 브릿지 수수료가 누적됨
  • "탈중앙" → 검증자·개발팀·재단·VC 집중 (Lido·Coinbase 등)
  • "글로벌 즉시" → 거래소 입출금 지연, 규제 차단 흔함

4.7 토큰화가 진짜로 독점적인 영역 — 3가지

기존 시스템이 정말 못하는 것은 다음 3가지로 좁혀진다. 이것들이 진짜 토큰화의 가치다.

1. 무허가 혁신 (Permissionless Innovation)

기존: 새 금융 서비스를 만들려면 → 은행·예탁원·금융위·증권사 등의 승인·계약·인프라 통합 필요. 몇 달~몇 년.
토큰화 (퍼블릭): 누구나 즉시 dApp을 배포 가능. 기존 토큰을 그대로 활용. DeFi가 폭발적으로 발전한 본질적 이유.

예: Uniswap은 단 며칠 만에 배포되어 수십억 달러 거래 시장을 형성. 기존 금융에서는 불가능한 속도

2. 자기 보관 (Self-Custody)

기존: 자산 보관은 항상 기관에 의존 (은행·증권사·예탁원). 기관이 동결·해킹·파산하면 손실 위험.
토큰화 (퍼블릭): 개인키 = 소유권. 어떤 기관도 동결·차단 불가 (이론상). FTX 사태 이후 "자기 보관"의 가치 재조명.

단점: 키 분실 = 영구 손실. 자기 책임이 어려운 일반 사용자에게는 부담

3. 조합 가능성 (Composability) — "Money Lego"

기존: A 은행 채권을 → B 증권사 옵션과 → C 펀드 ETF에 결합 = 거의 불가능. 각자 폐쇄 시스템.
토큰화 (퍼블릭): 레고 블록처럼 자유 조합. USDC를 Aave에 예치 → aUSDC 받음 → Curve LP에 → CRV 받음 → ... 단일 트랜잭션 가능.

이게 DeFi가 "money lego"라 불리는 이유. 무한 조합으로 새 금융 상품 창출 가능

4.8 도입 의사결정 — 3가지 질문

토큰화 도입 의사결정은 다음 3가지 질문에 답해야 한다. 모두 "예"가 아니라면 기존 시스템 개선이 더 합리적일 수 있다.

Q1. 무허가 혁신이 정말 필요한가?
    └─ NO  → 퍼미션드 DLT 또는 그냥 잘 만든 DB
    └─ YES → Q2로

Q2. 자기 보관이 정말 가치 있는가? (사용자 책임 감당 가능?)
    └─ NO  → 커스터디 서비스 + 기존 시스템
    └─ YES → Q3로

Q3. 다른 서비스와의 조합(Composability)이 핵심 가치인가?
    └─ NO  → 토큰화 효익이 작음 → 기존 시스템 개선 검토
    └─ YES → 토큰화 + 퍼블릭 블록체인 + 표준(ERC-20/3525 등)

4.9 퍼미션드 체인의 경우 — 가치는 더 좁아진다

위 분석은 주로 퍼블릭(permissionless) 블록체인 기준이다. 퍼미션드 체인(예: 한강, Hyperledger Fabric, R3 Corda, QBFT 기반 컨소시엄)을 선택하면 토큰화의 가치는 더욱 좁아진다. 각 가치가 어떻게 변하는지 객관적으로 살펴본다.

4.9.1 퍼미션드로 가면 사라지거나 약화되는 가치

가치퍼블릭퍼미션드이유
무허가 혁신✅❌운영 컨소시엄의 승인 없이 dApp 배포 불가
자기 보관✅⚠️보통 운영 기관(은행 등)이 관리. 한강도 은행 앱 안에서 운영
조합 가능성✅❌폐쇄적 → 외부 dApp이 들어올 수 없음
검열 저항⚠️ (약함)❌운영자가 트랜잭션 검열·자산 동결 가능
글로벌 접근✅❌승인된 참여자만 접근
외부 개발자 생태계✅❌접근 불가

4.9.2 그렇다면 퍼미션드 체인의 진짜 가치는?

퍼미션드 체인이 여전히 제공하는 (그리고 단일 DB가 제공하기 어려운) 가치:

  • 다자간 신뢰 모델: 한 기관이 아닌 여러 기관 공동 운영 → 단일 기관이 데이터 임의 조작 불가 (가장 중요한 가치)
  • 변조 저항 (감사 추적): 모든 노드가 같은 데이터 보유 + 합의 검증 → 사후 위변조 불가능
  • 합의된 단일 진실: 별도 reconciliation 절차 없이 모든 참여자가 같은 상태 공유
  • 표준화된 분산 인프라: 자체 구축보다 검증된 오픈소스(Fabric, Besu, Corda) 활용
  • 스마트 컨트랙트 호환성: 향후 외부 토큰화 자산(예: ERC-3525)과 연계 가능

4.9.3 그런데... "잘 설계된 분산 DB"로도 가능한가?

대부분의 가치가 다중 DBMS + 감사 로그 + 정합성 검증으로 어느 정도 구현 가능하다. 정직한 비교:

요건분산 DB로 구현 가능?퍼미션드 블록체인의 우위
다자간 합의 ⚠️ 가능 (2PC, Paxos, Raft) BFT는 비잔틴 노드(악의적 참여자) 견딤 — Paxos/Raft는 단순 장애만 견딤
변조 저항 ⚠️ 부분적 (감사 로그 + 해시 체인) 모든 노드 합의 검증으로 더 강력
스마트 컨트랙트 ✅ 가능 (Stored Procedure, 워크플로우 엔진) EVM 호환성 → 이더리움 생태계 도구·표준 활용
다자간 거버넌스 ⚠️ 별도 구축 필요 온체인 보팅 등 기본 제공
외부 토큰화 자산과 호환 ❌ 별도 인터페이스 구축 표준 토큰(ERC-20/3525 등) 즉시 호환

4.9.4 퍼미션드 체인이 적합한 경우

  • ✅ 다수 독립 기관이 공동 운영해야 (단일 기관 운영이면 그냥 DB)
  • ✅ 비잔틴 내성이 진짜 필요 (참가자 중 일부의 악의 가정)
  • ✅ 이미 컨소시엄이 형성되어 있고 기관 간 신뢰 부족
  • ✅ 변조 저항이 법적·규제적 핵심 요구사항
  • ✅ 향후 외부 토큰화 자산(CBDC·증권 등)과 연계 필요

4.9.5 퍼미션드 체인이 부적합한 경우

  • ❌ 단일 조직 내부 데이터 → 그냥 DB 또는 데이터 레이크
  • ❌ 성능이 핵심 → 중앙 DB가 BFT 대비 10~100배 빠름
  • ❌ 프라이버시가 절대 우선 → 채널·ZK 등 추가 복잡 설계 필요
  • ❌ 외부 개발자 생태계 활용 → 퍼블릭이 적합
  • ❌ 진정한 탈중앙화 필요 → 퍼블릭 외에는 의미 없음

4.9.6 한강·CBDC 컨소시엄에의 적용

한강 프로젝트의 퍼미션드 모델을 객관적으로 평가하면:

  • "왜 BOK-Wire+가 아닌가?": 이미 도매 결제 즉시성·법적 효력 확립되어 있음
  • "왜 분산 DB가 아닌가?": 7개 은행 + BoK 노드를 BFT로 운영해도, 잘 설계된 분산 DB와의 본질적 차이는 좁음
  • 실제 차별화 가치:
    1. 프로그래머블 머니 (청소년 바우처, 조건부 결제 등)
    2. 향후 토큰화 자산(ERC-3525, 토큰증권)과 표준 호환
    3. Agora 국경간 결제와의 연계 기반
    4. 비잔틴 내성으로 한 은행의 악의·장애가 시스템 전체 영향 차단

객관적 평가: 퍼미션드 CBDC의 가치는 "지금 결제를 더 효율적으로"보다 "향후 토큰화 자산·국경간 결제·프로그래머블 머니 시대를 위한 인프라"에 있다. 현재 효익만 비교하면 기존 시스템(BOK-Wire+ 등) 우위라는 회의론이 타당하며, 미래 옵션 가치를 어떻게 평가하느냐가 도입 정당화의 핵심이 된다.

4.9.7 어디서 더 효율적인가 — 다자간 조율 비용 절감 6가지

퍼미션드 체인의 본질적 unique함은 적지만, "다자간 조율(multi-party coordination) 비용"에서는 명백히 더 효율적이다. 이것이 도입 정당화의 진짜 근거다.

핵심 원칙: 단일 조직 내부에서는 블록체인이 비효율적이지만, 다수 독립 기관이 협력하는 환경에서는 진짜 효율적이다. 기존 시스템의 비용은 "다수 당사자가 같은 데이터를 같은 시점에 보고 합의하기 위한 조율 비용"에서 발생하며, 퍼미션드 블록체인이 이를 구조적으로 줄인다.
① Reconciliation(대사) 비용 제거
기존: A은행 DB ──┐
      B은행 DB ──┼─→ 매일 야간 reconciliation
      C은행 DB ──┤   → 불일치 발견 → 수동 조정
      D은행 DB ──┘   → 백오피스 인력·시간 비용

블록체인: 모든 노드가 항상 동일 상태 → reconciliation 자체 불필요
  • 한국 은행권 일별·월별 대사 업무 = 백오피스 인력의 큰 비중
  • SWIFT 메시지 reconciliation 오류 = 글로벌 연 수십억 달러 비용 추산
  • 가장 큰 효율 영역 — 한강·Agora의 핵심 매력 포인트
② 신뢰 설정 비용 감소
  • 기존: A 기관이 B 기관 데이터를 신뢰하려면 별도 검증·계약·중재자 필요. N개 기관 = N×(N-1)/2개 양자 협정 (7개 은행 = 21개)
  • 블록체인: 단일 합의 표준으로 신뢰 설정. 신규 기관 합류 시 한 번의 거버넌스 결정으로 종료
③ 다자 거버넌스 자동화
  • 기존: 검증자 추가·정책 변경 = 위원회 → 안건 → 합의 → 문서화 → 시스템 반영 (수주~수개월)
  • 블록체인: 온체인 보팅 + 스마트 컨트랙트로 수분~수시간에 자동화

한강 사례: 향후 참여 은행 확대 시 거버넌스 비용 차이가 큼

④ 다자 트랜잭션의 원자성(Atomicity)

기존: 2-Phase Commit·Saga 패턴 등으로 가능하나 복잡·취약. 부분 실패 시 보상 트랜잭션 필요.

블록체인: 한 트랜잭션이 모든 참여자 상태를 동시 갱신. DvP·PvP가 native 작동.

시나리오: 채권 토큰 + 예금토큰 동시 이전 (DvP)

기존: A 증권사 채권 → B 이전 → 별도 메시지 → B 은행이 A 은행에 송금
      → 양쪽 실패 시 reversal 복잡, T+2 결제일

블록체인: 단일 트랜잭션으로 채권 + 예금토큰 동시 이전
      → 둘 다 성공 or 둘 다 실패 (Atomic), T+0 즉시 결제
⑤ 외부 생태계 표준 호환
  • 기존: 새 자산·외부 시스템 연계 시 자체 인터페이스 구축. 토큰증권·CBDC·국제 결제망 연계 = 매번 별도 통합 개발
  • 블록체인 (EVM 호환): ERC-20·721·3525 등 검증된 표준 즉시 활용. Agora·다른 컨소시엄과 표준 인터페이스로 연결. 외부 개발 도구(Foundry·OpenZeppelin) 그대로 사용

이건 현재가 아닌 미래 옵션 가치가 큼 — RWA·STO·CBDC 생태계가 성숙할수록 효익 증가

⑥ 감사·규제 대응 비용 절감
  • 기존: 각 기관 감사인이 각자 데이터 검증 → 데이터 정합성 추가 검증. 규제기관 보고: 각 기관이 별도 형식으로 제출 → 통합·교차 검증 비용
  • 블록체인: 모든 노드가 동일 데이터·동일 검증 → 단일 진실 자동 보장. 규제기관이 노드 1개만 보면 모든 거래 검증 가능. 사후 위변조 사실상 불가능

4.9.9 효율적이지 않은 영역 (균형)

퍼미션드 블록체인이 더 비효율적인 영역도 분명히 있다:
  • ❌ 단순 처리 성능: 중앙 DB가 10~100배 빠름
  • ❌ 저장 효율: 모든 노드가 동일 데이터 → 저장 비용 N배
  • ❌ 단일 조직 내부 처리: 분산 자체가 오버헤드
  • ❌ 고도 프라이버시: 채널·ZK 등 복잡 설계 필요
  • ❌ 빠른 정책 변경: 한 기관이 단독으로 못 바꿈 (장점이자 단점)
  • ❌ 고객 응대 SLA: 합의 지연이 영향

4.9.10 한국은행·금융권 시사점

한강·Agora 같은 컨소시엄 환경에서 진짜 효율 가치:

효율 영역한강·Agora에서의 의미
Reconciliation 제거7개 시중은행 일별 대사 업무 큰 감소 (가장 큰 효익)
다자 거버넌스 자동화신규 은행 추가·정책 변경 비용 큰 절감
Atomic 다자 거래DvP·PvP 즉시 결제 (T+0 가능)
외부 표준 호환향후 STO·국경간 결제(Agora) 연계 옵션
감사·규제 대응금감원·BIS 보고 자동화

핵심: 한강 도입의 정당화는 "새로운 기능"이 아니라 "다자간 운영 비용 절감 + 미래 옵션 가치"에 있다.

최종 정리:
  • 토큰화의 8가지 가치 중 독점적인 것은 3가지(무허가 혁신·자기 보관·조합 가능성)뿐
  • 퍼미션드 체인은 이 3가지마저 대부분 잃음 → 남는 가치는 "다자간 신뢰 + 비잔틴 내성"으로 좁아짐
  • 그러나 다자간 조율 비용(Reconciliation·신뢰설정·거버넌스·Atomic 거래·외부 표준 호환·감사)에서는 진짜 효율적
  • 도입 정당화는 "새로운 기능"이 아니라 "운영 비용 절감 + 미래 옵션 가치"로 평가해야 정직
  • "왜 분산 DB가 아닌가?"에 대한 답 = "비잔틴 내성 + 표준 토큰 인프라 + 외부 생태계 호환"
  • 한강 같은 컨소시엄 CBDC는 현재 효익보다 미래 토큰화 인프라 옵션 가치로 평가하는 게 정직함

5. 한국 토큰증권(STO) 가이드라인

2023년 2월 금융위원회는 "토큰증권(ST)" 발행·유통 가이드라인을 발표했다. 현행 자본시장법 체계 내에서 분산원장 기술을 활용해 증권을 발행할 수 있도록 한 정책 변화.

  • 적용 대상: 자본시장법상 증권으로 분류되는 디지털자산
  • 발행 방식: 전자증권법 개정으로 분산원장에서의 증권 등록 허용 추진
  • 유통 인프라: 증권사·예탁결제원·KRX 등 기존 인프라와 연계
  • 혁신금융서비스(샌드박스): STO 실증사업 다수 진행 중

한국은행의 한강 프로젝트(예금토큰)와 결합되면 토큰증권 + CBDC + DvP 자동 결제가 단일 인프라에서 실현될 수 있다. 이것이 자산 토큰화의 궁극적 가치 — 결제와 자산 이전의 통합 — 이다.

📚 참고: 금융위 토큰증권 가이드라인 · Security Token Standard

한 줄 정리: 디지털화 = "기관의 전자 장부", 토큰화 = "본인이 직접 보유하는 프로그래머블 자산". 토큰화가 우월한 것이 아니라 각각의 강점이 다른 영역이며, 실제로는 두 시스템이 결합된 하이브리드 형태로 발전 중이다.

6.3 Terra/UST 붕괴 사례 — 알고리즘 스테이블코인의 한계

2022년 5월, 시가총액 약 600억 달러의 Terra/LUNA 생태계가 1주일 만에 99% 붕괴했다.

구조

  • UST: 알고리즘 스테이블코인 ($1 페그)
  • LUNA: 가치 흡수·차익거래용 토큰
  • 메커니즘: UST 가격이 $1 미만이면 → LUNA로 교환 가능 → 차익거래자가 UST 매수 → 가격 회복 (이론상)

붕괴 시나리오

  1. 대규모 UST 매도 발생 (2022.05.07)
  2. UST → LUNA 교환 발생 → LUNA 발행량 폭증
  3. LUNA 가격 하락 → 차익거래 매력 감소 → UST 추가 매도
  4. 피드백 루프 ("죽음의 나선")
  5. 1주일 만에 UST $0.10, LUNA 99% 손실

교훈

  • 알고리즘 스테이블코인은 신뢰가 깨지는 순간 회복 불가능
  • "무담보 가치 안정"은 게임이론적 가정에 의존, 극한 상황에서 무너짐
  • 이후 EU MiCA·미국 규제는 알고리즘 스테이블코인을 사실상 금지·제한

6.4 한국과 스테이블코인

  • 원화 스테이블코인 부재: KRW 페그 스테이블코인은 거의 없음 (KRWO 등 일부 시도)
  • 가상자산이용자보호법(2024.07)은 스테이블코인 발행을 명시적으로 규율하지 않음, 추후 입법 예정
  • 한국은행 한강의 예금토큰은 사실상 KRW 스테이블코인의 대안 — 다만 발행 주체가 상업은행
  • 은행 발행 스테이블코인: 일본 자금결제법 모델 (은행·신탁사만 발행)이 한국에서도 논의 중
왜 중요한가: 스테이블코인은 단순한 결제 도구가 아니라 "디지털 형태의 화폐"이며, CBDC와 직접 경쟁·보완 관계에 있다. 한강 예금토큰과 USDC·USDT의 본질적 차이는 "발행자가 누구인가 (상업은행 vs 민간 기업) + 어떤 자산을 담보로 하는가 (현금·국채)"이다.

7. DAO와 온체인 거버넌스

DAO(Decentralized Autonomous Organization)는 스마트 컨트랙트로 운영되는 의사결정·자금 관리 조직이다. 토큰 보유자가 투표로 의사결정에 참여한다.

7.1 DAO의 주요 유형

유형목적대표 사례
Protocol DAODeFi 프로토콜 파라미터 의사결정MakerDAO, Uniswap, Compound
Investment DAO공동 투자 결정 (벤처·NFT)The LAO, MetaCartel Ventures
Grants DAO생태계 보조금 분배Gitcoin, Optimism Citizens House
Service DAO전문가 조합 (개발·디자인)RaidGuild, DeveloperDAO
Social DAO커뮤니티 운영Friends With Benefits, Bored Ape
Single-Purpose DAO특정 목표 달성 후 해산ConstitutionDAO (미 헌법 경매 입찰, 실패 후 환불)

7.2 거버넌스 토큰의 작동 방식

// 단순화된 거버넌스 컨트랙트
contract Governor {
    mapping(address => uint256) public votingPower;
    mapping(uint256 => Proposal) public proposals;

    function propose(bytes calldata calldata_, string calldata description)
        external returns (uint256 proposalId);

    function castVote(uint256 proposalId, uint8 support) external;
    // support: 0=Against, 1=For, 2=Abstain

    function execute(uint256 proposalId) external;
    // 정족수 + 찬성 다수 충족 시 컨트랙트가 자동 실행
}

7.3 DAO의 한계와 비판

온체인 거버넌스의 현실적 문제점:
  • 낮은 참여율: 대부분의 DAO 투표 참여율 5~15% (전체 토큰 보유자 기준)
  • 고래(Whale) 지배: 상위 10개 주소가 의결권의 50%+ 보유하는 경우 흔함
  • 대리투표(Delegation) 집중: a16z·재단 등 대규모 위임 받은 소수가 사실상 결정
  • 투표 매수·뇌물: Convex·Aura처럼 vote-as-a-service 시장 형성
  • 의사결정 속도: 토론·투표·실행에 수일~수주 → 위기 대응 어려움
  • 법적 책임 불명확: DAO 구성원의 법적 책임 (Wyoming주 DAO LLC 등 일부 입법 시도)

7.4 주요 사고·교훈

  • The DAO 해킹 (2016): Reentrancy로 360만 ETH 탈취 → Ethereum 하드포크 → ETH/ETC 분리
  • Beanstalk 거버넌스 공격 (2022): 플래시론으로 의결권 확보 → 자금 탈취 (1.8억 달러)
  • Mango Markets (2022): 가격 조작 + 거버넌스 투표로 본인 행위 합법화 시도
  • Compound 7% 사고 (2021): 거버넌스 제안의 코드 버그로 8천만 달러 잘못 분배
시사점: DAO는 "탈중앙 의사결정"의 이상적 모델로 제시되지만, 실제로는 중앙화된 영향력자 + 낮은 참여 + 보안 취약이라는 현실적 문제를 안고 있다. 완전한 탈중앙 거버넌스보다 "투명한 의사결정 도구" 수준으로 평가하는 것이 객관적이다.

8. 컨트랙트 디자인 패턴

8.1 Proxy Upgrade Pattern

배포된 컨트랙트는 불변이 원칙이지만, 실무에서는 버그 수정·기능 추가가 필요하다. Proxy 패턴은 사용자가 보는 주소(Proxy)를 그대로 두고, 내부 구현(Implementation)만 교체한다.

패턴특징
Transparent Proxyadmin/user 함수 분리 (OpenZeppelin 표준)
UUPS (EIP-1822)업그레이드 로직을 구현체에 둠 (가스 효율)
Diamond (EIP-2535)여러 facet(구현)을 결합, 무제한 크기
Beacon Proxy여러 프록시가 단일 beacon 공유

8.2 Factory Pattern

새 컨트랙트를 동적으로 생성. CREATE2는 결정론적 주소 계산이 가능해 카운터팩추얼 배포에 사용.

// CREATE2: 배포 전에 주소 예측 가능
address computed = address(uint160(uint256(keccak256(abi.encodePacked(
    bytes1(0xff), factory, salt, keccak256(bytecode)
)))));

8.3 Pull over Push 패턴

돈을 직접 보내지 말고(push), 수신자가 가져가게 하라(pull). 수신자가 컨트랙트일 경우 revert로 전체 트랜잭션이 실패하는 사고를 막을 수 있다.

8.4 Checks-Effects-Interactions

Reentrancy 방지의 황금 규칙: 검사 → 상태 변경 → 외부 호출 순서를 지킬 것.

function withdraw(uint256 amount) external {
    // 1. Checks
    require(balances[msg.sender] >= amount, "Insufficient");

    // 2. Effects (외부 호출 전에 상태 갱신)
    balances[msg.sender] -= amount;

    // 3. Interactions (마지막에)
    (bool ok, ) = msg.sender.call{value: amount}("");
    require(ok, "Transfer failed");
}

  • 모든 함수에 적절한 access modifier
  • 0 주소·자기 자신 주소 입력 처리
  • 이벤트 누락 (인덱서 깨짐 위험)
  • 가격 oracle의 신뢰성과 지연 처리
  • 업그레이드 가능 컨트랙트의 storage layout 호환성
  • 가스 한도·외부 호출 실패 처리
  • 고정 소수점 산술의 정밀도 손실
  • 알려진 SWC ID(SWC Registry) 전수 확인
