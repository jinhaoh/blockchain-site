1. 블록체인이란?

블록체인(Blockchain)은 암호학적으로 연결된 블록의 순차적 자료구조를 분산 네트워크의 다수 노드가 공유하고, 합의 알고리즘으로 단일 상태에 도달하게 만드는 분산원장(Distributed Ledger) 기술이다. 중앙 신뢰 기관 없이도 변조 저항성(tamper-evidence), 이중지불 방지, 최종성(finality)을 제공한다.

📚 더 읽기: Bitcoin 백서 (Nakamoto, 2008) · Ethereum 백서 · 위키백과: 블록체인 · BIS Working Paper #880: Rise of CBDCs

핵심 명제: 블록체인은 단일 기술이 아니라 해시 함수 + 디지털 서명 + P2P 네트워크 + 합의 알고리즘의 공학적 조합이다. 각 구성요소의 트레이드오프를 이해해야 시스템을 올바르게 설계할 수 있다.
그림 1. 블록체인 스택 5계층 — 하위 계층은 상위 계층에 신뢰의 뿌리(root of trust)를 제공한다.

2. 블록 구조

각 블록은 헤더(Header)와 바디(Body)로 구성된다. 이전 블록의 해시를 헤더에 포함시켜 체인을 형성한다.

struct BlockHeader {
    int32_t  version;          // 4  bytes
    uint8_t  prevBlockHash[32]; // 32 bytes — 이전 블록의 SHA256d
    uint8_t  merkleRoot[32];    // 32 bytes — 트랜잭션 머클 루트
    uint32_t timestamp;        // 4  bytes — Unix epoch
    uint32_t bits;             // 4  bytes — 난이도 타깃 (compact)
    uint32_t nonce;            // 4  bytes — PoW 카운터
};

헤더의 어떤 비트라도 변경되면 SHA256(SHA256(header)) 결과가 완전히 달라지므로 이후 모든 블록의 prevBlockHash가 깨진다. 이것이 변조 저항성의 기술적 근거다.

3. 암호학적 해시 함수

SHA-256, Keccak-256(이더리움) 같은 해시 함수는 다음 세 가지 성질을 만족해야 한다:

📚 참고: NIST FIPS 180-4 (SHA-2) · Keccak Team (SHA-3) · Wikipedia: Cryptographic Hash · MDN: SubtleCrypto.digest()

  • 전상저항성(Pre-image resistance): h = H(x)에서 x를 역산하기 어려움
  • 2차 전상저항성(Second pre-image): H(x) = H(x')인 x' ≠ x 찾기 어려움
  • 충돌저항성(Collision resistance): H(x) = H(y)인 임의의 (x, y) 쌍 찾기 어려움
SHA-256 인터랙티브 데모
입력 1글자만 바꿔도 해시 전체가 완전히 달라집니다 (Avalanche Effect)

4. 머클 트리 (Merkle Tree)

블록에 포함된 수천 개의 트랜잭션을 단 32바이트의 머클 루트(Merkle Root)로 요약하는 이진 해시 트리. SPV(Simplified Payment Verification) 클라이언트는 전체 블록을 다운로드하지 않고도 O(log n) 크기의 머클 증명(Merkle Proof)으로 트랜잭션 포함 여부를 검증할 수 있다.

📚 참고: Wikipedia: Merkle Tree · Investopedia: Merkle Tree · Bitcoin 백서 §8 (SPV) · Ethereum Patricia Trie

Root = H(H12 ‖ H34)
H12 = H(H1 ‖ H2)
H34 = H(H3 ‖ H4)
H1 = H(Tx1)
H2 = H(Tx2)
H3 = H(Tx3)
H4 = H(Tx4)
// Merkle Proof 검증 (의사 코드)
function verifyProof(txHash, proof, root) {
  let computed = txHash;
  for (const { hash, position } of proof) {
    computed = position === 'left'
      ? sha256(hash + computed)
      : sha256(computed + hash);
  }
  return computed === root;
}

5. 합의 알고리즘

비잔틴 장군 문제(Byzantine Generals Problem)를 분산 환경에서 해결하기 위한 메커니즘.

📚 참고: Ethereum: PoW · Ethereum: PoS · Castro & Liskov: PBFT 원논문 · Tendermint BFT · Eyal & Sirer: Selfish Mining

그림 2. PoW vs PoS — 보안의 원천이 "에너지"에서 "자본"으로 이동
알고리즘 대표 체인 최종성 비잔틴 내성 특징
PoW Proof of Work Bitcoin, Litecoin 확률적 ≤ 50% 높은 보안성, 에너지 소비 큼
PoS Proof of Stake Ethereum 2.0, Cardano 확률적 / 결정적 ≤ 33% 에너지 효율적, 스테이커 집중 위험
BFT PBFT / Tendermint Cosmos, Hyperledger 즉시 결정적 ≤ 33% 고성능, 노드 수 제한 (퍼미션드)
DPoS Delegated PoS EOS, TRON 결정적 ≤ 33% 대표 노드 위임, 탈중앙화 약화

6. 주요 블록체인 체인별 특징

체인은 합의 알고리즘, 데이터 모델, VM, 거버넌스, 목적에 따라 매우 다른 설계를 갖는다. 동일한 "블록체인"이지만 트레이드오프가 다르므로, 유스케이스에 맞는 선택이 필요하다.

7. 트랜잭션 생명주기

  1. 생성: 클라이언트가 트랜잭션 구성 → privKey로 ECDSA/EdDSA 서명
  2. 전파: P2P gossip 프로토콜로 메모리풀(Mempool)에 진입
  3. 검증: 노드들이 서명·잔액·논스(nonce) 검증
  4. 블록 포함: 채굴자/검증자가 우선순위(수수료 기준)로 선택해 블록에 포함
  5. 합의: 합의 알고리즘에 따라 블록이 확정
  6. 최종성 도달: 비트코인 6 confirm (≈ 1시간), 이더리움 finality gadget

8. 스마트 컨트랙트

블록체인 위에서 실행되는 결정론적 프로그램. 이더리움의 EVM(Ethereum Virtual Machine)이 대표적이며, Solidity로 작성된 코드는 바이트코드로 컴파일되어 모든 노드에서 동일하게 실행된다.

📚 참고: Solidity 공식 · Ethereum: EVM · OpenZeppelin 표준 컨트랙트 · SWC Registry (보안 약점 분류) · Slither 정적 분석 · Foundry Book 

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract SimpleToken {
    mapping(address => uint256) public balanceOf;
    uint256 public totalSupply;

    event Transfer(address indexed from, address indexed to, uint256 value);

    constructor(uint256 _initial) {
        totalSupply = _initial;
        balanceOf[msg.sender] = _initial;
    }

    function transfer(address to, uint256 amount) external returns (bool) {
        require(balanceOf[msg.sender] >= amount, "Insufficient");
        unchecked {
            balanceOf[msg.sender] -= amount;
            balanceOf[to] += amount;
        }
        emit Transfer(msg.sender, to, amount);
        return true;
    }
}
보안 함정: Reentrancy (DAO 해킹), Integer Overflow, Front-running (MEV), Oracle 조작, Access Control 누락 등은 모두 실제 수억 달러 규모의 손실로 이어진 사례가 있다. Slither, Mythril, Foundry의 fuzzing 등 정적/동적 분석 도구 사용은 필수다.

11. 크로스체인 브릿지 (Cross-Chain Bridges)

12.2 Project 한강 (Hangang) — 한국은행 CBDC 활용성 테스트

한국은행이 2024년 발표하고 2025년 본격 진행한 CBDC 기반 예금토큰 실거래 테스트. 이전의 모의실험(2021~2022)과 달리, 실제 일반 이용자가 참여하는 real-world utility test로 설계되었다.

주관한국은행 + 금융위원회 + 금융감독원
참여 은행국내 7개 시중은행 (KB, 신한, 하나, 우리, NH농협, IBK기업, BNK부산 등)

활용 사례 가맹점 QR 결제, 청소년 급식비·교육바우처 등 프로그래머블 머니 시범

     │

특징적인 기술·정책 포인트:

  • 퍼미션드 DLT: 일반 이용자는 지갑 없이 은행 앱을 통해 접근 (UX는 기존 모바일뱅킹과 유사)
  • 프로그래머블 머니: 청소년 바우처처럼 사용처·대상을 제약할 수 있는 조건부 결제
  • 예금토큰 모델: 일반 이용자가 직접 보유하는 것은 CBDC가 아닌 상업은행 예금토큰 → 예금자보호 대상이며 기존 금융시스템과 정합성 유지
  • 오프라인 결제·프라이버시는 본 단계에서 제한적이며 후속 단계 검토 사항

13. x402: HTTP 기반 결제 프로토콜

x402는 1997년 HTTP/1.1 명세에서 예약만 되어 있던 HTTP 402 Payment Required 상태 코드를 실제로 작동하게 만든 오픈 결제 프로토콜이다. 2025년 Coinbase가 공개했으며, AI 에이전트와 머신 간 결제(machine-to-machine), API 종량 과금을 1차 타깃으로 한다.

📚 참고: x402.org 공식 · coinbase/x402 GitHub · EIP-3009 (transferWithAuthorization) · EIP-712 (Typed Data) · RFC 7231 §6.5.2 (HTTP 402) · Circle USDC

13.1 왜 지금인가?

  • AI 에이전트 경제: 자율 에이전트가 API·콘텐츠·연산 자원을 마이크로결제로 소비하는 시나리오 증가
  • 스테이블코인의 정착: USDC 등 결제 적합성·결제 최종성·낮은 수수료를 갖춘 자산 등장
  • 전통 결제망의 한계: 신용카드 최소 결제 단위(보통 $0.30+α), KYC, 30~120일 지연 정산
  • 표준의 부재: API 키 + 청구서 모델이 에이전트 워크플로우에 부적합

13.3 EIP-3009: 가스리스 결제의 핵심

x402가 사용자 친화적인 이유는 EIP-3009 transferWithAuthorization에 있다. 사용자(에이전트)는 USDC 자체만 보유하면 되고, ETH 가스비는 facilitator가 대납한다. 서명은 EIP-712 typed data로 만들어진다. 

// USDC 컨트랙트의 EIP-3009 인터페이스
function transferWithAuthorization(
    address from,
    address to,
    uint256 value,
    uint256 validAfter,
    uint256 validBefore,
    bytes32 nonce,
    uint8 v, bytes32 r, bytes32 s   // EIP-712 서명
) external;

// 클라이언트(에이전트) 측 의사 코드
const auth = {
  from: agentWallet.address,
  to: serverPaymentAddress,
  value: parseUnits("0.001", 6),     // USDC 6 decimals
  validAfter: 0,
  validBefore: now() + 600,          // 10분 유효
  nonce: randomBytes32(),
};
const signature = await agentWallet.signTypedData(
  USDC_DOMAIN, EIP3009_TYPES, auth
);
// → X-PAYMENT 헤더로 전송

13.4 facilitator 패턴

서버가 매 요청마다 직접 온체인 검증·정산을 하면 RPC 비용·지연이 발생하므로, facilitator라는 중개자가 권장된다. 서버는 facilitator에 서명을 보내 검증과 정산을 위임받고, 응답을 받은 뒤 리소스를 반환한다.

구성요소역할
Client / Agent지불 의사 결정 + EIP-3009 서명
Resource Server402 응답으로 가격 통지 + 결제 후 자원 반환
Facilitator서명 검증, 온체인 정산 대납, txHash 반환
Settlement NetworkBase, Ethereum, Solana 등 결제 자산이 발행된 체인

13.5 시나리오: 에이전트가 자율적으로 API 결제

// LLM 에이전트가 유료 검색 API를 호출하는 의사 코드
async function callPaidAPI(url, agentWallet) {
  let res = await fetch(url);

  if (res.status === 402) {
    const requirement = JSON.parse(res.headers.get('X-PAYMENT-REQUIRED'));

    // 에이전트의 정책 (예산, 자산, 화이트리스트) 검사
    if (!agentPolicy.canPay(requirement)) {
      throw new Error("Payment policy rejected");
    }

    const payment = await signX402Payment(requirement, agentWallet);
    res = await fetch(url, {
      headers: { 'X-PAYMENT': payment }
    });
  }
  return res.json();
}

13.6 한계와 열린 과제

  • 규제·세무 모호성: 자율 에이전트의 결제 주체는 누구인가? 한국은 가상자산이용자보호법 적용 검토 필요
  • 가격 발견 메커니즘 부재: 정적 가격 책정 → 동적 경매·번들링은 후속 표준 영역
  • 리플레이·재정산 공격: nonce + validBefore에 의존 → 클라이언트 구현 결함 시 이중지불 위험
  • facilitator 신뢰: 단일 facilitator는 검열 위험, 다중화 시 비용 증가
  • 비-EVM 체인 확장: 현재 USDC on Base가 1차 타깃, Solana/L2 확장은 진행 중
  • 금융권 시각: CBDC·예금토큰을 결제 자산으로 사용할 수 있는가? 한강 모델과 결합 시 새로운 통화정책 함의 발생
한국은행 관점의 함의: x402가 가정하는 결제 자산은 현재 스테이블코인이지만, 프로토콜 자체는 자산 중립적이다. 한강 프로젝트의 예금토큰을 x402 결제 자산으로 사용하는 시나리오는, M2M·에이전트 경제에서 은행화폐(commercial bank money)의 단위성 유지를 실험할 수 있는 흥미로운 정책 실험이 될 수 있다.
